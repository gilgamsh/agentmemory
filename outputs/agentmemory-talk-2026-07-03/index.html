agentmemory: 让 coding agent 记住项目上下文

从自动捕获、混合检索到跨 agent 协作，把“记忆”变成可运营的项目基础设施。

v0.9.27 iii-engine primitives MCP / REST / Viewer

开场先定调：这不是一个“多存一点聊天记录”的项目，而是 coding agent 的上下文运行时。不逐行读源码，只讲路径级源码：哪些入口捕获信息，信息怎么进入索引，agent 后续怎么通过 MCP、REST 或 viewer 找回来。最后会明确边界，哪些能力默认零 LLM，哪些是 opt-in。

Problem

问题不是“记不住”，而是上下文不可运营

项目上下文散落在提示词、聊天、文件和人的脑子里；越到后期，agent 越像一次性进程。

重复解释

架构约束、测试习惯、线上事故、用户偏好反复出现在 prompt 里。

旧决策丢失

“为什么这里这么写”常常需要翻 commit、issue、聊天记录，还不一定找得到。

静态规则过期

AGENTS.md 或 CLAUDE.md 适合硬规则，不适合持续吸收每次会话里的事实。

叙事起点上下文需要生命周期，而不是更长 prompt

这里强调 pain：agent 的问题不是完全没有记忆，很多工具已经有静态规则文件或聊天历史。真正的问题是这些上下文没有运营能力：没有来源、没有过期、没有检索策略、也不能跨工具复用。agentmemory 要解决的是“把上下文当成系统状态管理”，而不是单纯把一堆文本塞回窗口。

Mental Model

一句话模型：Capture → Index → Recall → Coordinate

agentmemory 是一个本地记忆服务：自动观察 agent 会话，建索引，在下一次需要时以受控预算返回上下文。

01

Capture

hooks、REST、MCP 把 prompts、工具调用、提交信息和显式记忆写入同一条链路。

02

Index

BM25、向量和关系图围绕 observation、memory、lesson 建可检索结构。

03

Recall

根据问题、文件、会话、commit 或 token budget 返回压缩后的证据。

04

Coordinate

actions、leases、signals、slots、handoff 支撑多 agent 和跨会话工作流。

Engine

iii 心智模型：Worker / Function / Trigger

agentmemory 不绕过 iii-engine；新能力就是注册函数、注册触发器，再通过 sdk.trigger() 组合。

W

Worker

agentmemory 进程启动后连接 iii-engine，注册所有 mem::*、api::*、mcp::* 函数。

F

Function

业务能力以函数暴露，例如 mem::observe、mem::smart-search、mem::remember。

T

Trigger

HTTP、stream、cron、event 把外部请求映射到函数，统一进入 iii 的状态和调用模型。

sdk.registerFunction("mem::observe", handler)
sdk.registerTrigger({ type: "http", function_id: "api::observe" })
await sdk.trigger({ function_id: "mem::smart-search", payload })
架构规则所有能力通过 registerFunction / registerTrigger / sdk.trigger()

这一页只讲三个原语。Worker 是进程，Function 是能力，Trigger 是入口。重要的是一致性：agentmemory 不是 REST 里一套、MCP 里一套、SQLite 又一套，最终都会落到 iii 的函数调用和 StateModule。这样做的好处是可观测、可组合、可跨入口复用，也解释了为什么 AGENTS 里强调不要绕过 iii-engine。

两个执行策略

需要把上下文写回 stdout 的 hooks 使用 await fetch(... AbortSignal.timeout)。
只做捕获的 hooks 使用 fire-and-forget，并用短 setTimeout(...).unref() 保证进程退出。
默认捕获不需要 LLM；自动压缩和上下文注入分别通过环境变量 opt-in。
plugin/hooks/hooks.jsonsrc/hooks/* / agentmemory-hooks skill

这页讲设计取舍。agent 不能因为记忆系统慢就卡住下一步，所以遥测类 hook 要把请求发出去后尽快退出。只有 SessionStart、PreToolUse、PreCompact 这类需要把回忆内容注入 stdout 的脚本才等待响应，而且必须有 timeout。这样默认路径是轻量捕获，昂贵的 LLM 摘要和自动注入都是显式开启。

默认路径的含义

不设置 AGENTMEMORY_AUTO_COMPRESS 时，每条观察仍会生成 zero-LLM 的压缩文本，因此 recall 和 BM25 可以工作。

图像、vision embedding、LLM summary、context injection 都是独立开关，不是基础捕获的前置条件。

src/functions/observe.tssrc/functions/compress-synthetic.ts / src/functions/search.ts

这页是第一个源码路径，但不逐行讲。核心顺序是：验证、去重、脱敏、写 KV、推送 stream、更新 session，再选择压缩策略。默认 synthetic compression 很关键：即使没有 API key，也不会只存一坨不可检索 raw JSON，而是生成标题和叙述用于 BM25 与向量索引。这样产品默认可用，成本能力另行开启。

Recall Path

检索链路：BM25 + Vector + Graph + RRF

mem::smart-search 是常用入口：多路召回、加权融合、按 session 去集中化，再按 token budget 暴露细节。

keyword

BM25

文件名、函数名、错误消息、显式术语命中稳定，默认无需 API key。

semantic

Vector

本地默认 embedding 支撑改写、近义、跨表达检索；可持久化并做维度校验。

relations

Graph

可选关系图扩展概念与实体，适合跨 session 的关联发现。

combinedScore =
bm25Weight × 1/(60 + bm25Rank)
+ vectorWeight × 1/(60 + vectorRank)
+ graphWeight × 1/(60 + graphRank)

返回不是越多越好

过量召回后按 agent scope 和 project 过滤。
按 session 做 diversification，避免一个会话淹没结果。
memory_recall 支持 token_budget，展开用 expandIds 渐进披露。
src/state/hybrid-search.tssrc/functions/smart-search.ts / src/mcp/server.ts

检索页解释为什么不是单一向量库。代码里 HybridSearch 用 RRF 形式融合 BM25、vector 和 graph rank，默认权重来自配置。然后做 session diversification，减少同一会话刷屏。MCP 层的 memory_smart_search 先返回 compact 结果，需要更多上下文时再 expandIds；memory_recall 面向直接注入，支持 token_budget。

Lifecycle

生命周期与治理：让记忆变少、变准、可删除

长期有用的不是“永远保存”，而是捕获、压缩、合并、验证、遗忘的闭环。

整理

consolidate 把工作记忆、episodic、semantic、procedural 分层；lessons 保存可复用经验。

保留

slots 维护小而稳定的项目/全局记忆；retention 和 auto-forget 控制增长。

治理

audit、verify、governance_delete 让状态变更可追踪，可解释，也可删除。

默认边界

基础捕获与 BM25 检索不需要 LLM。LLM compression、graph extraction、context injection、vision embedding 都是配置开关。

协作边界

多 agent 可以通过 AGENT_ID 标记写入，用 AGENTMEMORY_AGENT_SCOPE 控制召回隔离。

src/functions/*README: configuration, governance, agent scope

生命周期页强调边界和治理。记忆系统最怕两件事：无限增长和不可解释。agentmemory 的回答是分层整理、slot 化、保留策略、审计和验证。还要明确 opt-in：LLM summary 会花 token，context injection 会消耗上下文，graph extraction 也有开关。默认路径是轻量、本地、可检索。

Workflow

落到我们的工作流：三条实践建议

让记忆成为工程习惯，而不是偶尔想起来才用的搜索框。

01

开工先 recall

碰到架构、历史决策、跨文件修改、测试习惯时，先查过去会话，不要让 agent 重新猜。

memory_recall(query)
02

重要结论要 remember

设计选择、迁移坑、用户偏好、复盘结论值得显式保存，后续自动捕获再补充证据。

memory_save(content, concepts)
03

交接看 session 与 commit

新会话用 handoff 接上下文；追溯代码用 commit-context 找回当时 agent 的工作轨迹。

handoff / commit-context
结尾Capture → Index → Recall → Coordinate

结尾回到可执行建议。第一，开工先 recall，特别是遇到历史和跨文件问题。第二，重要结论要显式 remember，因为自动捕获是材料，显式记忆是判断。第三，交接和追溯用 session、handoff、commit-context，把“为什么这样写”从口头知识变成可查询证据。最后重复四段模型收束：Capture、Index、Recall、Coordinate。
