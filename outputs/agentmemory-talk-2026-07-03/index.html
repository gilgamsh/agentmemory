agentmemory

从事件流到长期知识：agentmemory 的记录、索引、检索与总结机制

memory lifecycle retrieval evaluation v0.9.27

开场先把叙事从“工具集合”切到“记忆生命周期”。今天不把重点放在 MCP、REST 或 viewer 的界面数量，而是解释一个 coding agent 的事件流如何变成长期知识：先记录什么，再压缩成可索引单元，之后如何融合检索，什么时候总结和整理，最后用 retrieval benchmark 说明效果和边界。

Problem

问题不是上下文短，而是经验不可积累

把更多历史塞进 prompt 不能解决长期记忆的三个核心约束：不可检索、不可验证、不可演化。

不可积累

一次会话里的尝试、错误、决策和文件变更，默认不会沉淀为下一次可复用的知识。

不可检索

旧上下文散落在聊天、终端输出、commit 和文件里；agent 只能重新猜，不能按证据找回。

不可验证

没有 observation id、session id、来源路径和删除记录，记忆就会变成不可审计的软提示。

问题定义上下文需要生命周期，而不是更长 prompt

这一页先建立科研听众熟悉的问题形式：我们不是在问“窗口能不能更长”，而是在问经验是否能成为可查询、可验证、可更新的状态。长上下文只解决短期容量；长期 agent 需要的是从事件到索引、从索引到检索、从检索到组织的闭环。后面的每个机制都对应这里的一个约束。

Lifecycle Model

Capture → Compress → Index → Retrieve → Summarize → Consolidate

agentmemory 把“记忆”拆成可实现、可测量、可治理的六个阶段。

01

Capture

从 prompt、工具调用、错误、文件变更和显式保存中形成事件流。

02

Compress

把 raw event 转成标题、叙述、概念、文件和置信度。

03

Index

对同一观察建立 keyword、semantic 和 relation 入口。

04

Retrieve

按问题融合召回，控制 session 多样性和 token budget。

05

Summarize

把一个 session 压成结构化摘要，保留决策和文件线索。

06

Consolidate

跨观察聚合成长期 memory、lesson、slot 和治理记录。

系统问题

每个阶段都是一个边界：输入是什么、产物是什么、失败时怎么降级、是否需要 LLM、是否可审计。

评测问题

最容易先评测的是 retrieval：gold session 是否出现在 top-k。summary 和 consolidation 需要额外质量指标。

mental modelcapture / compression / indexing / retrieval / organization

这页给出全场主线。六个阶段不是营销词，而是系统边界：Capture 负责材料，Compress 负责把材料变成可索引单元，Index 决定检索入口，Retrieve 决定如何拿回来，Summarize 是会话级压缩，Consolidate 是长期知识组织。科研听众可以把它看成 retrieval system 加 memory lifecycle 的组合。

What To Store

什么适合记，什么不适合直接记

长期记忆不是全量日志。好记忆应该有来源、能被检索、未来会改变 agent 行为。

适合进入 observation

用户意图和任务边界：prompt intent、目标、约束。
工具动作和结果：命令、搜索、读写文件、失败信息。
文件编辑和决策：改了哪里、为什么改、替代方案。
显式记忆：偏好、架构事实、工作流、已确认的经验。

不适合原样长期化

secret、token、私密片段：先做隐私过滤，不把泄漏固化。
噪声日志和长输出：除非包含错误、指标或决策证据。
无来源偏好：没有 session 或观察来源，后续很难验证。
临时猜测：应保留为低置信 observation，而不是强记忆。
recording policysrc/functions/privacy.ts / src/functions/observe.ts

这一页解释为什么不能把所有东西都叫记忆。agentmemory 的基础材料是 observation，但长期有用的是带来源的观察：用户想做什么、工具做了什么、失败在哪里、最后做了什么决定。相反，secret、噪声日志和没有来源的偏好会污染系统。这里的标准不是“文本是否出现过”，而是它未来能否作为证据被找回和验证。

Index Maintenance

索引维护：写入即更新，失败可重建

BM25 是同步默认路径；vector 是 guarded best-effort；graph 来自长期结构和关系函数。

observe.ts synthetic observation kv.set 到 KV.observations 后立即 getSearchIndex().add，再调用 vectorIndexAddGuarded。
compress.ts LLM compressed obs mem::compress 生成更丰富的 CompressedObservation，同样写 KV、BM25、vector。
remember.ts explicit Memory mem::remember 将 Memory 转成 observation 形状进入 BM25 和 vector。
rebuildIndex(kv) boot fallback 索引为空时从 KV.memories 和每个 session 的 observations 重建。
BM25: getSearchIndex().add(observation)
vector: vectorIndexAddGuarded(id, sessionId, title + narrative)
rebuild: clear BM25 + vector → batch reload from KV

可靠性策略

vector 写入会检查 provider 是否存在、embedding 维度是否匹配、embed 是否抛错。失败只记录 warn 并返回 false，不阻塞 observation 或 memory 保存；BM25 和 KV 仍然可用，重启后 rebuild 会补齐可补的索引。

src/functions/search.tssrc/functions/observe.ts / compress.ts / remember.ts

这一页讲维护而不是只讲三路检索概念。observe、compress 和 remember 都有 live-write 路径：先写 KV，再更新 BM25，再尝试 vector。vectorIndexAddGuarded 是关键可靠性边界：没有 provider、维度不匹配、embed 网络失败都会 soft-fail，不让保存失败。rebuildIndex 是恢复路径：索引为空时从 KV.memories 和所有 session observations 重建，同时清理 stale vector。graph 不在这里强行承诺实时更新，它来自 graph 和 relation 相关函数，在检索时作为补充流使用。

Hybrid Retrieval

算法核心：RRF_K=60，按可用检索流归一化

BM25、vector、graph 先各自产生 rank，再用 reciprocal-rank fusion 合并。

BM25

limit × 2

路径、符号、错误码、命令参数走 lexical rank；没有 embedding provider 时仍能工作。

Vector

limit × 2

查询 embedding 失败则空结果；top 5 vector hits 还会触发 graph expansion。

Graph

entities + expansion

先从 query 抽实体做 searchByEntities，再从 top vector chunks 做一跳扩展。

combinedScore = Σ effectiveWeight(stream) × 1 / (60 + rank)
default weights: BM25 0.4, vector 0.6, graph 0.3
unavailable streams get weight 0; remaining weights renormalize

后处理不是装饰

合并后先按 session diversify，默认每个 session 最多 3 条；retrievalDepth 至少 20；可选 rerank 只处理前 20 条，失败时回退到 RRF 顺序。

src/state/hybrid-search.tssrc/functions/smart-search.ts / src/mcp/server.ts

这一页讲算法而不是接口。tripleStreamSearch 对 BM25 和 vector 都取 limit 乘以 2 的候选，graph 由实体检索和 top vector observation 的一跳扩展组成。融合时 RRF_K 是 60；如果 vector 或 graph 不可用，它们的权重变成 0，剩余权重重新归一化。之后做 session diversification，避免同一会话占满 top-k。rerank 是可选小窗口步骤，失败时直接返回 RRF 结果，所以检索不会依赖 reranker 可用性。

这里把总结当作可靠性问题，而不是简单调用一次模型。小会话走单次总结；大会话默认每 400 条 observation 一块，并发 6 个 chunk 调用。每个 chunk 解析失败、provider error 或网络失败都会 retry once；如果失败块超过 50%，函数直接报 too_many_chunks_skipped。reduce 阶段只合并解析成功的 partial summaries。最终响应还会 strip XML wrapper、schema validation，并计算 qualityScore，所以 summary 是结构化产物，而不是不可控文本。

Consolidation 回答“怎么从材料变成知识”。代码默认要求至少 10 条候选 observation，只保留 importance 大于等于 5 的观察，按 concept 分组，并且每组至少 3 条才进入模型。每组只取 importance 最高的 8 条，每次运行最多 10 次 LLM 调用。Memory 类型被限制在六类：pattern、preference、architecture、bug、workflow、fact。关键是 sourceObservationIds、version 和 project scope：长期记忆不是无来源结论，而是可追溯、可演化、可限制作用域的结构化记录。

Long-Term Structure

“梳理”不是只总结，而是分层放置知识

不同知识进入不同生命周期层：热的保留、稳的固化、错的修正、敏感的删除。

层次不是装饰

summary 给会话可读骨架，memory 给长期事实和模式，lesson 给可复用经验，slot 给小而稳定的上下文，audit/verify/forget 给治理闭环。

这页解释“梳理”的含义。总结只是其中一层：它压缩一个 session，但不会自动变成长期原则。Memory 是跨观察的长期结论，Lesson 是可强化经验，Slot 是少量稳定上下文，Governance 层让这些状态可验证、可审计、可删除。这样长期记忆不会只是越来越大的文本文件，而是有生命周期位置的知识结构。

删除不是只删一行：还要删除访问日志、搜索索引和向量索引，否则旧证据会从检索路径复活。

这页是长期记忆和普通 prompt cache 的关键区别。Memory 不是无来源的自然语言结论；sourceObservationIds 让它能回到具体 observation。mem::verify 会把 memory 的 citation 找回来，并显示 session project、status、timestamp 等来源信息。审计层记录谁做了 remember、evolve、compress、forget 或 delete。删除路径也不是只删 KV 行，还要清理 access log、BM25、vector index，并 flush 持久化，避免已经删除的记忆从索引快照里重新出现。

Limits & Cost

边界：检索系统有效，不等于记忆永远正确

科研分享里需要把可测能力、成本来源和失败模式讲清楚。

retrieval

可评测的是找回证据

LongMemEval-S 的 R@5 衡量 gold session 是否进入 top-k；reader 是否正确使用证据是另一个问题。

LLM cost

默认路径不烧模型

capture 默认 synthetic compression；LLM 成本主要来自 auto-compress、summarize、consolidate 和可选 rerank。

freshness

索引和 KV 可能短暂不一致

vector 写失败会降级；删除路径需要 flush index save；rebuildIndex 是恢复和补齐机制。

quality

总结和整理依赖生成质量

summary 有 schema 和 quality score；consolidation 有 evidence cap 和 audit，但仍需要 verify 和 forget 闭环。

boundary conditionsretrieval ≠ reader correctness ≠ task success

这一页防止过度声明。agentmemory 的强项是把历史事件变成可检索证据，并且这个 retrieval 能被 benchmark。它不保证下游 agent 一定读懂证据，也不保证 LLM 生成的 summary 或 consolidation 永远正确。成本上，默认 capture 不调用 LLM；真正的模型成本来自 opt-in auto-compress、summarize、consolidate 和可选 rerank。可靠性上，vector failure 是可降级的，删除需要同步索引，长期结论需要 verify 和 forget 闭环。

Takeaway

何时该记，何时该搜，何时该总结

给科研听众的结论：agentmemory 是一个可评测的 retrieval + lifecycle 系统。

01

该记

当信息未来会改变 agent 行为，并且能绑定到来源时，记录成 observation 或显式 memory。

capture / remember
02

该搜

当问题依赖历史决策、文件变更、错误处理或跨 session 事实时，先 retrieve，再生成。

retrieve top-k evidence
03

该总结

当单个 session 过长或需要交接时做 summary；当重复模式出现时再 consolidate 成长期记忆。

summarize / consolidate
closingCapture → Compress → Index → Retrieve → Summarize → Consolidate

结尾把系统变成操作原则。该记的是有来源、会影响未来行为的事件；该搜的是历史证据，而不是让 agent 重新猜；该总结的是长会话和交接上下文；该 consolidation 的是多次出现的模式。最终 takeaway 是：agentmemory 不是单点工具，而是一个可以做 retrieval 评测、也能做生命周期治理的长期记忆系统。
