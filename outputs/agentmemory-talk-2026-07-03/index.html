agentmemory

从事件证据到可检索记忆：agentmemory 的检索与记忆形成机制

memory lifecycle retrieval evaluation v0.9.27

开场先把 agentmemory 放在科研问题里：它不是一个接口清单，而是在研究 coding agent 的历史如何变成可评测的证据，再进一步形成长期记忆。今天主线只有两条：第一，事件怎样被压缩成可以被检索系统命中的 evidence unit；第二，多次 evidence 怎样被整理成更稳定的 memory。这样后面的设计、指标和边界都能围绕 recall 与 memory formation 来讲。

Problem

问题不是上下文短，而是经验不可积累

把更多历史塞进 prompt 不能解决长期记忆的三个核心约束：不可积累、不可检索、不可追踪。

不可积累

一次会话里的尝试、错误、决策和文件变更，默认不会沉淀为下一次可复用的知识。

不可检索

旧上下文散落在聊天、终端输出、commit 和文件里；agent 只能重新grep，不能按证据找回。

不可追踪

没有 observation id、session id 和来源路径，就无法定位证据，也无法知道长期结论来自哪段历史。

问题定义long context is storage, not accumulated memory

这一页保留原来更直观的问题表述：长期 agent 的痛点不是单纯窗口短，而是经验没有积累机制，也没有按证据找回的能力。第三点用“不可追踪”来替代抽象的“不可验证”：如果历史没有 observation id、session id 和来源路径，我们就不知道 retrieval 的 gold evidence 是什么，也无法回到支撑长期结论的那段历史。后面的 compress、retrieve 和 consolidate 都是在补这些追踪边界。

Lifecycle Model

五种操作，对应不同产物边界

capture、compress、retrieve、summarize、consolidate 不是操作同一种“记忆”，而是记忆周期里的不同转换。

关键区分

compress 让单条事件可以被索引和召回。
summarize 让一个 session 可以被人或 agent 快速浏览。
consolidate 从多条证据里归纳稳定模式。
mental model下一页：observation / index / summary / memory / graph 关系

这页专门解决术语混淆。这里不是说有五种“记忆粒度”，而是说生命周期里有五种不同产物边界。capture 产生带来源的事件证据；compress 把单条事件变成检索系统能排序的单元；retrieve 返回 top-k evidence；summarize 面向一个 session，产物是会话摘要；consolidate 才是长期记忆形成，它需要跨多条 observation 找重复模式，再生成有来源的长期 Memory。下一页会把 observation、检索索引、summary、memory 和 knowledge graph 的生成关系摊开，强调哪些默认自动、哪些需要额外整理。边界分清之后，评测也更清楚：retrieval 先评证据能不能回来，summary 和 consolidation 要另设质量标准。

这一页把可返回内容、检索索引和派生 store 分开。底层仍然是同一个 iii-engine StateModule 管理的 KV / SQLite state，不是三套数据库；区别在逻辑角色。recall 最终主要返回 observation 和长期 memory。BM25 和 vector 是排序信号，relation store 是从 observations 抽出的实体关系索引，retrieval 可以用它做 graph expansion 来扩展候选；这些信号可以组合排序，但它们本身不是新的 memory。summary 写入 KV.summaries，服务 handoff、context、recap 和后续 consolidation，不直接进入 recall top-k，也不直接写 BM25/vector 或 graph。

What To Store

记录边界：先问它未来能否作为证据

长期记忆不是全量日志。适合保留的信息应能被检索、能回到来源，并且会改变未来行为。

适合进入 observation

用户意图和任务边界：prompt intent、目标、约束。
工具动作和结果：命令、搜索、读写文件、错误、指标。
文件编辑和决策：改了哪里、为什么改、哪些方案被排除。
已确认事实：偏好、架构约束、工作流、可复用经验。

不适合直接变长期事实

噪声日志和长输出：除非包含错误、指标或决策证据。
无来源偏好：没有 session 或 observation 来源，后续很难核对。
未确认结论：先保留为观察证据，不要直接写成长期事实。
瞬时状态：进度提示、一次性 UI 状态、不会影响未来决策的文本。
recording policyobservation evidence before long-term claim

这一页回答“捕获什么才有研究价值”。agentmemory 不是把 transcript 原样堆起来，而是优先保存未来可能被查询的问题证据：任务边界、工具结果、文件改动、失败原因和已确认事实。未确认结论也可以留下，但应该停留在 observation 层，等更多证据出现后再 consolidate。系统边界仍会处理敏感字段和坏输入，但主线不是合规工程，而是让材料能成为未来 recall 与记忆形成的可靠依据。

Retrieval Signals

一个 observation，三种检索入口

BM25、vector、graph 不是工程冗余，而是分别覆盖 lexical、semantic 和 relational evidence。

BM25

精确词与代码符号

文件路径、函数名、错误码、命令参数、测试名这类字符串证据最适合 lexical matching。

Vector

语义相似

当 query 和 observation 不共享关键词时，embedding 帮助召回 paraphrase、意图相近的证据。

Graph

实体与关系

session、文件、概念和相邻证据形成扩展入口，用于跨步骤或跨会话问题。

检索信号的研究含义

同一条 observation 同时暴露 lexical、semantic 和 relational views，实验才能比较哪类问题依赖哪类证据。没有向量时仍可用 keyword evidence；向量只是提高语义覆盖。

src/functions/search.tssrc/functions/observe.ts / compress.ts / remember.ts

这一页从“索引维护”改成“检索信号构造”。科研上我们关心的是不同信号解决什么召回问题：BM25 擅长路径、符号和错误码；vector 擅长语义相似和换一种说法的查询；graph 擅长把 query entity 扩展到相关 session、文件或概念。agentmemory 对同一 observation 建多种入口，是为了让 recall 不只依赖一种相似度。没有 embedding 时，keyword evidence 仍然可以工作；有 embedding 时，语义覆盖会更好。

Hybrid Retrieval

四步把候选变成 top-k evidence

BM25、vector、graph 先各自产生候选，再用 RRF_K=60 的 rank fusion 得到可注入的证据列表。

01 recall 召回候选 BM25 和 vector 各自返回候选 observation；graph 提供关系候选。
02 expand 实体扩展 从 query 和高分语义候选抽实体，补入相邻文件、session、概念证据。
03 fuse rank fusion RRF_K=60 表示高排名证据加分更明显，低排名证据仍有小贡献。
04 diversify session 多样性 合并后限制单个 session 占比，避免 top-k 被同一段历史淹没。

为什么不用单一路径

coding-agent 历史里同一个问题可能表现为文件名、错误输出、自然语言意图或跨会话模式。hybrid retrieval 让这些证据流都能投票，最后返回可排序的 observation evidence。

src/state/hybrid-search.tssrc/functions/smart-search.ts

这一页讲 retrieval algorithm 的核心机制，不讲接口差异。第一步各流召回候选：BM25 抓精确词，vector 抓语义相近，graph 抓关系扩展。第二步利用实体和高分候选做一跳扩展，补足跨 session 或跨文件线索。第三步用 reciprocal-rank fusion 合并，RRF_K=60 的直觉是：排名靠前的候选贡献更大，但多个弱信号一致时也能累积。第四步做 session diversification，避免 top-k 全来自一个 session。这样返回的不是一段长上下文，而是一组可评测证据。

先返回 compact top-k，再按需展开 observation

检索系统先回答“哪些证据可能相关”，上下文注入再决定“哪些证据需要全文”。

Consolidation

consolidate：从多条 evidence 形成长期 Memory

它不是再写一个 session 摘要，而是把跨 session 的重复证据归纳为有来源、可演化的长期结论。

Memory

和 summarize 的区别

summarize 压缩一个 session，产物帮助理解那段历史；consolidate 跨多段历史找重复模式，产物会影响未来行为。

长期 Memory 的可信度来自证据集合，而不是一次生成文本本身。

src/functions/consolidate.tssrc/types.ts: Memory

这一页讲 memory formation 的机制。consolidation 的输入是一组候选 observations，不是一段 transcript。系统先按概念、文件、错误类型或任务语境分组，再从每组选择最能支持结论的 top evidence，之后才让 LLM 综合成长期 Memory。也要提醒听众：Memory 不是每条 observation 写入时自动生成；它来自显式 remember 或 consolidation 这类整理流程。输出要保留 sourceObservationIds，并且可以随新证据演化为新版本。这个流程的研究意义在于：长期记忆不是一次性生成的摘要，而是由多条可追溯证据支撑的归纳结论。

研究含义

可追溯来源让 consolidation 的输出可以被人工或 evaluator 复核；可撤销能力让长期记忆不会因为一次错误归纳而永久污染后续行为。

这一页把治理压到最小必要闭环。长期 Memory 必须保存 sourceObservationIds，因为没有来源就无法判断它是不是合理归纳。verify 的作用是把结论带回原始 observation 和 session，让人或 evaluator 可以检查证据是否支持结论。forget 和 delete 则处理另一类问题：当来源不该继续使用、结论过期或归纳错误时，系统必须能撤销它。对科研叙事来说，治理不是工程附录，而是让 memory formation 可复核的条件。

Limits & Cost

边界：retrieval recall ≠ reader correctness ≠ task success

科研分享里需要把可测能力、生成质量和模型成本分开声明。

retrieval

R@k 只衡量证据是否回来

LongMemEval-S 的 R@5 是 gold session 是否进入 top-k，不是最终回答正确率。

reader

reader 仍可能误读证据

证据进上下文后，agent 是否正确引用、推理和执行，是另一层评测。

formation

长期记忆仍需复核

consolidation 能归纳模式，但生成式结论仍要依赖来源核对和撤销机制。

cost

默认 capture 不调用 LLM

模型成本主要来自 auto-compress、summarize、consolidate 和可选 rerank。

boundary conditionsretrieval ≠ reader correctness ≠ task success

这一页是边界声明。agentmemory 当前最强、最可量化的是 retrieval：历史证据能不能被找回来。reader correctness 是下一层问题，即 agent 拿到证据后是否读懂并正确使用。task success 又更远，还受到代码执行、测试覆盖和外部环境影响。成本也要分开：默认 capture 使用启发式压缩，不调用 LLM；只有 auto-compress、summarize、consolidate 和可选 rerank 这些生成或重排序步骤才产生模型成本。这样声明可以避免把 retrieval 指标夸大成端到端智能。

Takeaway

何时该记，何时该搜，何时该形成长期记忆

agentmemory 的科研主线：把历史切成证据，用检索找回证据，再从重复证据中形成 memory。

01

该记

当事件未来会改变 agent 行为，并且能绑定到 session 与来源时，记录成 observation。

capture evidence
02

该搜

当问题依赖历史决策、文件变更、错误处理或跨 session 事实时，先 retrieve top-k evidence。

retrieve before reasoning
03

该形成记忆

当多次 evidence 指向同一模式时，再 consolidate 成带来源、可演化的长期 Memory。

consolidate repeated evidence
closingCapture → Compress → Retrieve → Summarize → Consolidate

结尾回到三条操作原则。第一，记的对象不是所有文本，而是未来可能改变行为的 evidence。第二，遇到历史相关问题时应该先搜证据，再让 agent 推理，而不是让它重新猜。第三，长期记忆不应来自单次事件，而应来自多条 observation 的重复模式，并且保留 sourceObservationIds。用这个视角看，agentmemory 是一个 retrieval 加 memory formation 系统：它能把 coding agent 的历史变成可评测的 recall 问题，也能把重复证据整理成可复核的长期知识。
